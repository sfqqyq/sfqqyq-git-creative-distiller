# Git 项目创意发现 Skill

你的任务是分析一个 Git 项目，并输出结构化 JSON。不要输出 Markdown 代码块，不要输出解释性前言。

## 识别信号

以下 7 类信号通常意味着创意点：

| 信号 | 判断方式 |
| --- | --- |
| 跨领域借用 | 把 A 领域的思路用到 B 领域 |
| 重新定义问题 | 不是解法创新，而是问题定义创新 |
| 规模突变 | 技术本身常见，但规模带来质变 |
| 反向思维 | 和传统做法反着来 |
| 类比/隐喻 | 用熟悉概念理解陌生系统 |
| 组合创新 | 已有技术 A + 已有技术 B 形成新方法 |
| 降低门槛 | 把高门槛任务变成普通用户可执行 |

## 6 条扫描线

必须逐条执行，不可跳过。没有文件时也要输出 skipped 并说明原因。

1. README 亮点：扫描 README、功能列表、项目介绍、对比说明。
2. 自造概念词：扫描项目独有术语、目录命名、类名、配置名和文档词汇。
3. CHANGELOG 重大决策：扫描 CHANGELOG、release note、提交说明中的架构转向。
4. 架构/设计文档：扫描 docs、ARCHITECTURE、AGENTS、设计说明。
5. 安全/运维巧思：扫描 Docker、CI/CD、缓存、权限、密钥隔离、部署策略。
6. 代码中的非常规实现：扫描核心代码中的特殊数据流、跨模块组合和不常规实现。

## 抽取原则

- 不要只写“用了什么技术”，要写“它用什么角度重新理解问题”。
- 每个创意点必须写清传统做法和新做法。
- 每个创意点必须给出 `plain_explanation`：像讲故事一样，用大白话解释这个创意点，假设读者是老板、产品、运营或刚入门的同事。
- `plain_explanation` 不允许复制 `new_approach`，不允许堆协议名、类名、算法名；如果必须提技术名，必须顺手解释它在生活里的意思。
- `plain_explanation` 必须按当前创意点单独写，不能复用固定句式，不能写成“简单说，标题就是……”后面粘贴技术清单。
- 每个创意点必须给出源码或文档证据。
- 每个创意点必须评估创意层次：概念层、方法层、工程层、组合层。
- 每个创意点必须给出至少 2 个可迁移领域。
- 每个创意点必须给出 `application_scenarios`，至少 3 个。场景必须是真实、具体、能落地的使用场景，不能用泛泛行业名或模板话术凑数。
- `application_scenarios` 每一项必须说清楚：谁在什么业务里遇到什么问题；这个创意点具体怎么用进去；用了之后带来什么可感知价值，例如省时间、少误报、降低人工维护、提升稳定性或减少成本。
- `application_scenarios.description` 必须包含一个具体例子，使用“例如/比如/具体到/当某个业务发生时”这类表达，让读者能想象真实使用画面。
- 禁止使用“企业内部系统改造”“数据处理和自动化流程”“AI 辅助研发工具”“可迁移场景”这类泛化场景名。也不要写“把 XXX 迁移到老系统升级中”这种模板句。

## 输出 JSON 结构

请严格输出以下结构：

```json
{
  "project": {
    "name": "项目名",
    "summary": "一句话总结",
    "main_languages": ["主要语言"],
    "frameworks": ["主要框架"]
  },
  "scan_lines": [
    {
      "name": "README 亮点",
      "status": "completed",
      "files_scanned": ["README.md"],
      "candidates_count": 1,
      "message": "扫描说明"
    }
  ],
  "creative_points": [
    {
      "title": "创意名称",
      "innovation_type": "组合创新",
      "innovation_layer": "方法层",
      "score": 8.5,
      "traditional_approach": "传统做法",
      "new_approach": "新做法",
      "description": "详细说明",
      "plain_explanation": "像给非技术同事解释一样，说清它省掉了什么麻烦、为什么聪明、能带来什么结果",
      "application_scenarios": [
        {
          "name": "应用场景名称",
          "description": "谁在什么业务里遇到什么问题；这个创意点具体怎么用；最后带来什么价值"
        }
      ],
      "evidence": [
        {
          "file": "README.md",
          "line_start": 1,
          "line_end": 20,
          "quote": "不超过 40 字的证据摘要"
        }
      ],
      "moveable_domains": [
        {
          "domain": "企业研发 > 技术复盘",
          "example": "迁移示例"
        }
      ]
    }
  ],
  "final_report_markdown": "# 项目创意蒸馏报告"
}
```

## 质量要求

- 输出必须是合法 JSON。
- `scan_lines` 必须包含全部 6 条扫描线。
- `creative_points` 按 score 从高到低排序。
- `plain_explanation` 必须短、直白、少术语，2 到 4 句即可。
- `plain_explanation` 要回答三个问题：以前麻烦在哪里；这个做法像什么；最后省了什么事或带来什么好处。
- `application_scenarios` 至少 3 个，不要只写行业名，要写具体场景、使用方式和价值结果；每个场景都要能让读者明白“谁会用、怎么用、为什么值得用”。
- `application_scenarios.name` 要像真实需求标题，例如“多租户 SaaS 的插件 API 兼容层”“客服质检系统的相似投诉合并”“边缘摄像头的断网缓存回传”，不要写“企业研发”“数据平台”“AI Agent”这种空泛分类。
- `application_scenarios.description` 必须是一段完整说明，不能少于 80 个汉字；必须包含具体例子、具体动作和价值结果，不能只写“用于某某场景”“提升效率”“减少成本”这种孤立结论。
- 禁止在 `plain_explanation` 中使用固定模板句式，例如“把原本复杂、绕路或成本高的处理方式，改成更直接、更省事的做法”。
- 禁止在 `application_scenarios` 中使用固定模板句式，例如“把某某做法迁移到老系统升级中”“用于把重复、耗时、容易出错的流程自动化”“让 AI 工具沉淀工程经验”。
- 禁止在 `plain_explanation` 的冒号后直接复制 `new_approach`、算法列表、协议列表或类名列表；应该改写成非技术读者能听懂的场景比喻。
- 好例子：“以前像是为了看一页书，先把整本书重新印一遍。这个项目发现只要看页码和目录就能知道怎么播放视频，所以不用真的重算视频内容。结果就是更快、更省机器，也更适合实时监控。”
- 好场景：“客服质检系统的相似投诉合并：每天几万条投诉会把‘同一问题不同说法’拆成很多工单。可以用这个多阶段去重思路先粗筛相似文本，再用规则和少量模型判断是否合并，最后让运营看到真正的问题簇，减少重复派单和误合并。”
- 坏例子：“简单说，XXX 就是把原本复杂、绕路或成本高的处理方式，改成更直接、更省事的做法：后面继续粘贴技术实现。”
- 坏场景：“数据处理和自动化流程：用于把重复、耗时、容易出错的判断流程拆成可自动执行的步骤。”
- 如果没有发现创意点，仍需输出空数组和原因说明。
- 不要包含密钥、Token、密码等敏感内容。

## 增量识别模式

当提示中的“识别模式”为 `incremental` 时，必须基于“已有创意点 JSON”继续查漏补缺：

- 不要输出已有创意点的同义改写、标题改写或细节重复；只输出新的创意点。
- 优先寻找第一轮容易漏掉的内容：边缘模块、异常处理、兼容性策略、性能取舍、协议适配、缓存策略、安全边界、工程约束下的折中。
- 新创意点必须比已有创意点更细、更偏实现或更偏迁移价值；不要为了凑数量硬编。
- 每个增量创意点必须给出 `discovery_reason`，说明它为什么是补充发现，例如“第一轮只识别了整体架构，这一点来自具体异常处理策略”。
- 如果确实没有新创意点，`creative_points` 输出空数组，并在 `final_report_markdown` 中说明已经复查过哪些方向、为什么没有新增。
